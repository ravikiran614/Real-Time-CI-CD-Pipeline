// 01 · Overview

What Is This?

A production-grade, event-driven CI/CD pipeline designed for real-time deployments with built-in failure recovery, auto-rollback, and continuous feedback loops. Built to handle MLOps-scale workloads — not just standard web apps.

🔄

Closed-Loop System

Not linear — detects, deploys, monitors, patches, and re-deploys autonomously.

⚡

Event-Driven Core

Kafka decouples deploy triggers from execution for async, replayable deployments.

🔁

Auto-Healing

Health check failures trigger automatic rollback with default log capture.

📡

Live Feedback

SSE streams test results and deploy status to dashboard in real-time.

// 05 · Key Differentiator

Loop Architecture

Most CI/CD pipelines are linear — they end at deployment. This pipeline is circular.

Deploy → Monitor → Detect Change → Fix → Feedback → Build (again) └──── Rollback if unhealthy ────────────────┘
✅Self-heals on deploy failure via Auto-Rollback to last stable state
✅Continuously monitors post-deploy with Grafana + PagerDuty alerts
✅Merges small fixes without manual intervention using Git Auto-Merge
✅Streams all pipeline activity back for full observability via SSE
// 06 · Design Decisions

Why These Choices?

Why Kafka for prod deploy?

Kafka decouples the deploy trigger from the deploy execution — enabling async, event-sourced deployments that can be replayed or paused without pipeline restarts. Critical for MLOps-scale workloads.

Why SSE alongside Jest?

SSE streams test results in real-time to the dashboard as tests run — giving devs live feedback instead of waiting for a full run to complete. Reduces context-switch and feedback lag.

Why the "Change Detected?" gate?

Prevents unnecessary redeployments on zero-diff pushes (e.g., doc-only commits), routing them to monitoring-only mode instead. Saves compute and reduces deploy noise.

Why K8s for staging deploy?

Rolling deployments with liveness + readiness probes allow zero-downtime validation. If a pod fails health checks, K8s auto-reverts — which feeds directly into our rollback loop.

// 07 · Roadmap

Known Limitations & Future Work

  • Circuit breaker on Auto-Rollback loop — add max retry count to prevent infinite rollback cycles
  • Secrets management layer — integrate HashiCorp Vault or AWS Secrets Manager
  • Multi-environment matrix — extend to dev / UAT / canary environments
  • SBOM generation — add software bill of materials at build stage for supply chain security
  • Stream Feedback schema — formalize what gets streamed (logs vs metrics vs alerts)
  • Distributed tracing — integrate OpenTelemetry across all pipeline stages
